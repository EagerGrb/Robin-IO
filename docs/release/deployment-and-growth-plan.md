# Deployment And Growth Plan

Date: 2026-05-18

This document describes how to take the project from local code to a public npm release, then build adoption.

## 0. Product Positioning

Do not position the project as the fastest CSV parser. The stronger market position is:

> A cross-runtime TypeScript import/export runtime for large files, progress, validation, errors, batching, and plugins.

Short English tagline:

> Not just a parser. A full import/export runtime.

Primary users:

- SaaS teams building CSV import/export workflows.
- Internal tools and admin platforms.
- Browser apps that need large-file preview/export.
- Node services that need structured import pipelines with validation and error reporting.

## 1. Repository Strategy

Use GitHub as the source of truth.

Recommended setup:

1. Create a GitHub organization or repo owner account.
2. Create a public repository, for example:

   ```text
   github.com/<org>/io
   ```

3. Protect `main`.
4. Require pull requests into `main`.
5. Require CI before merge.
6. Use squash merge for normal feature PRs.
7. Use release commits generated by Changesets for version bumps.

Recommended branch model:

| Branch      | Purpose                             |
| ----------- | ----------------------------------- |
| `main`      | Always releasable                   |
| `feature/*` | Feature work                        |
| `fix/*`     | Bug fixes                           |
| `release/*` | Optional release preparation branch |

Minimum branch protection:

- Require `.github/workflows/ci.yml`.
- Require review before merge once there are collaborators.
- Block force-push to `main`.
- Require linear history if you want a clean release log.

## 2. Package Naming Decision

The current packages use the `@robbin-io/*` npm scope:

```text
@robbin-io/core
@robbin-io/node
@robbin-io/browser
@robbin-io/codec-csv
@robbin-io/codec-jsonl
...
```

Before publishing, confirm that you control the `@robbin-io` npm organization/scope. If not, rename every package to a scope you own, for example:

```text
@your-org/io-core
@your-org/io-node
@your-org/io-browser
```

or:

```text
@your-org/core
@your-org/node
@your-org/browser
```

Recommendation: use a distinctive owned scope such as `@robin-io/*` or a company/project scope. `@robbin-io/*` is short but likely hard to control and hard to search.

Run release packaging checks:

## 4. GitHub Actions

The repository already has:

```text
.github/workflows/ci.yml
.github/workflows/release.yml
```

CI should run on every PR and push to `main`:

```bash
npm run verify
npm run smoke:browser-matrix
```

Release should be manual for the first few releases:

1. Open GitHub Actions.
2. Run `Release`.
3. Keep `publish=false` for rehearsal.
4. If all gates pass, run again with `publish=true`.

## 5. npm Account And Publishing

Use npm Trusted Publishing when possible. It uses GitHub Actions OIDC, avoids long-lived npm tokens, and supports npm provenance.

Setup steps:

1. Create or choose the npm organization/scope.
2. Give your npm user owner/admin rights for the packages.
3. In npm package settings, configure Trusted Publishing for the GitHub repository and release workflow.
4. Keep `permissions.id-token: write` in `.github/workflows/release.yml`.
5. Keep `npm publish --provenance` in the publish script.

Fallback if Trusted Publishing is unavailable:

1. Create an npm automation token.
2. Add it to GitHub repository secrets as `NPM_TOKEN`.
3. Keep `NODE_AUTH_TOKEN: ${{ secrets.NPM_TOKEN }}` in the publish job.

Current publish script:

```bash
npm run release:publish
```

It publishes every non-private package under `packages/*` in dependency order and uses `--provenance` outside dry-run mode.

## 6. Versioning With Changesets

For every user-visible change:

```bash
npm run changeset
```

Choose semver level:

| Change                          | Version bump |
| ------------------------------- | ------------ |
| Bug fix                         | patch        |
| New backward-compatible feature | minor        |
| Breaking API change             | major        |

Before release:

```bash
npm run changeset:status
npm run changeset:version
npm run verify
```

Commit the version and changelog changes:

```bash
git add .
git commit -m "Version packages"
git push origin main
```

Then run the release workflow.

## 7. First Public Release Path

Recommended order:

1. Rename scope if needed.
2. Fix README encoding and polish the first page.
3. Add a minimal landing section:

   ```text
   Why this exists
   Install
   Node CSV -> JSONL example
   Browser CSV preview example
   Performance snapshot
   Package list
   ```

4. Run:

   ```bash
   npm run verify
   npm run bench
   npm run bench:stress
   npm run release:publish:dry-run
   ```

5. Create a changeset.
6. Run `npm run changeset:version`.
7. Merge to `main`.
8. Run `Release` with `publish=false`.
9. Run `Release` with `publish=true`.
10. Verify:

    ```bash
    npm view <package-name> version
    npm install <package-name>
    ```

## 8. Post-Publish Smoke Test

Create a temporary consumer project:

```bash
mkdir io-smoke-consumer
cd io-smoke-consumer
npm init -y
npm install @your-scope/core @your-scope/node @your-scope/codec-csv @your-scope/codec-jsonl
```

Create a tiny CSV -> JSONL script and run it with Node 24.

Also test browser packages through a Vite app:

```bash
npm create vite@latest io-browser-smoke
```

## 10. How To Build Awareness

### Week 1: Proof

Publish:

- GitHub repo.
- npm packages.
- A short blog post: "Building a TypeScript import/export runtime that works in Node and the browser".
- A performance note: "1M rows CSV -> JSONL in about 15s on Node 24".

Channels:

- Hacker News "Show HN".
- Reddit: `r/typescript`, `r/node`, `r/javascript`.
- X/Twitter thread.
- LinkedIn post if targeting SaaS/internal tools.
- Dev.to or personal blog.

### Week 2: Examples

Ship examples people can clone:

- Node CSV -> JSONL.
- Browser CSV upload preview.
- Browser JSON/CSV download.
- Validation + dead-letter example.

Make examples visual and copy-pasteable.

### Week 3: Integrations

Add one or two recognizable integrations:

- Zod validation guide.
- React browser upload example.
- Next.js server route import example.
- S3 or database connector design preview.

### Week 4: Trust

Add:

- Comparison page: parser vs import/export runtime.
- Stability matrix.
- Performance methodology.
- Security notes for CSV formula escaping.
- Roadmap.

## 11. Launch Messaging

Use this angle:

> CSV parsers are easy. Production imports are not. This project gives TypeScript apps a full streaming import/export runtime with progress, validation, errors, batching, Node/browser support, and plugin boundaries.

Avoid saying:

```text
The fastest CSV parser.
```

Say instead:

```text
A production-oriented import/export runtime for TypeScript.
```

## 13. Release Cadence

Before public adoption:

- Release often, about 1-2 times per week.
- Keep versions below `1.0.0`.
- Mark unstable APIs clearly.

Suggested roadmap:

| Version | Goal                                       |
| ------- | ------------------------------------------ |
| `0.1.x` | Public MVP, docs, examples                 |
| `0.2.x` | Performance and progress semantics cleanup |
| `0.3.x` | Browser UX examples and worker docs        |
| `0.4.x` | Plugin API hardening                       |
| `1.0.0` | Stable core API and release policy         |
